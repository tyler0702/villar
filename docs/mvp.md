# villar MVP仕様書 v6（最終確定）

> 「AIが書いたmdを、人間が読める形に再構成する」
> — 読む専用。元ファイルは一切変更しない。

---

## MVPで証明する核（2つだけ）

```
核1: セクションカードで「読みやすくなる」
核2: TL;DR + Mermaid変換で「理解しやすくなる」
```

---

## 非機能要件（プロダクトの前提）

| 要件 | 内容 |
|---|---|
| **ローカル完結** | 外部サーバーへの送信なし。ローカルフォルダのみ対象 |
| **元ファイル非破壊** | 元のMarkdownファイルを編集・上書きしない。表示上の再構成のみ行う |
| **読む専用** | 編集機能は一切持たない |

---

## 実装優先順位

```
1. H2カード
2. フォーカスモード
3. TL;DR
4. Rust watch
5. Mermaid直線変換
6. 自動折りたたみ
```

---

## フェーズ定義

### Phase 1 — 読書体験の基盤

| 機能 | 詳細 |
|---|---|
| フォルダ選択 | ダイアログでローカルフォルダを指定 |
| ファイル一覧 | 配下の `.md` を左サイドバーに表示 |
| Markdownレンダリング | remark/rehype で高品質表示 |
| **H2ベースのカード表示** | H2単位で分割・めくる体験 |
| **H2なし文書の救済** | H2が存在しない場合はH1直下の段落数ベースで暫定分割 |
| **フォーカスモード** | アクティブカード100%・他30% opacity |
| テーマ | ライト / ダーク |

**この段階で証明する価値：** スクロール地獄が消える

---

### Phase 2 — 理解支援レイヤー

| 機能 | 詳細 |
|---|---|
| **TL;DRカード** | H2先頭に自動生成（ルールベース） |
| Rust側ファイル監視 | 保存→即再レンダリング（debounce込み） |
| 自動折りたたみ | 長リスト/コードブロックを折りたたむ（優先度低め） |

**この段階で証明する価値：** 本文を読まずに章の概要を把握できる

#### TL;DR生成ロジック

```
概要：H2配下の最初のpタグ（先頭1〜2文）
ポイント：箇条書きの先頭3件
キーワード：太字テキスト（**xxx**）を抽出
結論：「結論」「まとめ」「つまり」で始まる行
```

#### TL;DR非表示条件（v6修正）

以下のいずれかを満たす場合、TL;DRカードを非表示にする。

```
・概要候補（最初のpタグ）が取れない
・抽出文字数が50文字未満
・概要・ポイント・結論のうち、有効な要素が1つ未満
```

**全構成要素を必須にしない。TL;DRはある程度欠けていても成立する。**
箇条書きや太字がない文書でも、概要だけで表示できればよい。

---

### Phase 3 — Mermaid変換（最小構成）

| 機能 | 詳細 |
|---|---|
| **flowchart直線構造のみ変換** | 全ノードが直線条件を満たす場合のみステップUIへ変換 |
| **フォールバック優先（二段階）** | 変換不能→元図表示→元図失敗→元テキスト表示 |
| 変換 ↔ 元図 切替 | ワンクリックで元Mermaid確認可能 |
| WebWorker + LRUキャッシュ | パフォーマンス確保 |

**この段階で証明する価値：** Mermaidが図ではなく意味として読める

#### Mermaid変換判定ロジック（構造条件ベース）

```
flowchartの構造情報を解析し、
全ノードの入出次数が以下を満たす場合のみステップUIへ変換する：

・全ノードの出次数 ≤ 1（分岐なし）
・全ノードの入次数 ≤ 1（合流なし）
・ループなし

→ 条件を満たす: ステップUI変換
→ 条件を満たさない: 元のMermaid図にフォールバック（判定失敗の表示なし）
```

**目的は構文木を扱うことではなく、直線構造を見抜くこと。**

---

## 信頼性ポリシー

### フォールバック優先

```
変換に自信がない → 必ず元のMermaidを表示する
TL;DR構成要素が不足 → 非表示（空白）
```

### 可逆性

- 変換後UIからワンクリックで元Mermaid確認
- 元Markdownの該当箇所へ戻れる

### エラー表示方針（v6修正・二段階）

| ステップ | 条件 | 表示 |
|---|---|---|
| 1 | 変換解析を試みる | — |
| 2 | 変換不能・条件不満足 | 元のMermaid図で表示を試みる |
| 3 | 元図レンダリングも失敗 | 「表示できません」＋元テキスト表示 |

変換解析の失敗と描画の失敗は別。元図で通るケースがあるため二段階にする。

---

## パフォーマンス予算（v6修正・計測条件追加）

**計測前提条件：**
- ファイルサイズ：100KB程度のmd
- Mermaidブロック：3個以内
- 実行環境：一般的な開発用ノートPC

| 指標 | 目標値 |
|---|---|
| 初回表示 | 500ms以内 |
| カード遷移 | 100ms以内 |
| ファイル更新反映 | 300ms以内 |
| Mermaid変換タイムアウト | 1秒 |

---

## 観測用ログ（内部ログ）

MVPだからこそ計測する。改善の軸を作るために必須。

| 計測項目 | 用途 |
|---|---|
| 開かれたファイル数・頻度 | 使われているか確認 |
| TL;DR生成成功率 | ルールベースの精度把握 |
| Mermaid変換成功率 | flowchart直線判定の精度 |
| フォールバック率 | 変換失敗の頻度 |
| 平均レンダリング時間 | パフォーマンス予算の達成確認 |

外部送信なし。ローカルログファイルに記録するのみ。

## 技術スタック

| レイヤー | 選定 | 備考 |
|---|---|---|
| フレームワーク | Tauri | 軽量・ローカル特化 |
| フロントエンド | React + TypeScript | 型安全 |
| 仮想スクロール | react-virtual | **大量カード時に導入。初期は不要** |
| Markdown | remark + rehype | plugin設計を最初から |
| Mermaid | mermaid.js + WebWorker + LRUキャッシュ | パフォーマンス対策 |
| ファイル監視 | Rust側（Tauri）+ debounce | chokidarより安定 |
| スタイル | Tailwind CSS | 高速開発 |

### remark pipeline

```
remark-parse
  └─ remark-tldr        // TL;DRカード生成（カスタム）
  └─ remark-section     // H2セクション分割 + H2なし救済（カスタム）
  └─ remark-collapse    // 自動折りたたみ（カスタム・Phase 2後半）
  └─ remark-rehype
  └─ rehype-mermaid     // Mermaid変換（カスタム・WebWorker）
  └─ rehype-highlight
  └─ rehype-stringify
```

## 実装時の制約（Claude Code向け）

実装時に以下を必ず守る。勝手に広げない。

```
・H2以外の動的分割は実装しない（H2なし救済のみ例外）
・Mermaid変換は直線flowchartだけ
・変換不能時は元図→元テキストの二段階フォールバック
・TL;DRはルールベースのみ（AI連携コードは書かない）
・react-virtualは初期実装に含めない
・機能追加の提案を勝手にしない
・元ファイルへの書き込みは一切しない
```
